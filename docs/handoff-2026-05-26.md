### 2. Activate Tier 3 (DBNet on-device) — needs model file

Follow [`docs/dbnet-conversion.md`](dbnet-conversion.md):

1. `wget` PaddleOCR `ch_PP-OCRv4_det_infer.tar` and unpack
2. `paddle2onnx` (Paddle → ONNX, opset 13)
3. `onnxsim` + `onnx2tf` (ONNX → TFLite, NHWC)
4. Copy `model_float32.tflite` → `assets/models/dbnet_paddleocr.tflite`
5. Add the asset entry to `pubspec.yaml` under `flutter: assets:`
6. Rebuild — `[DbnetTextDetector] model loaded` should appear in logcat

DBNet runs as a safety-net 5th pass over the ML Kit 4-pass result: for
every region DBNet finds that no ML Kit block already covers
(IoU > 0.3), the code crops the capture and runs ML Kit's Latin
recognizer on the crop. Logged as `[CameraService] DBNet filled in N
region(s)`.

### 4. Tune the weakness threshold

The composite ML Kit weakness score
([camera_screen.dart `_mlkitWeakness`](../lib/features/translate/screens/camera_screen.dart))
fires Vision fallback when ≥ 0.5. Each fallback emits a
`vision_fallback` analytics event with the computed `weakness` value.
Once enough real captures are logged:

- Many `weakness > 0.5` events that ML Kit ACTUALLY had right →
  raise threshold (less Vision spend).
- Captures missing text but never triggering fallback → lower threshold
  or fold in additional signals (edge-coverage; see "Improving live
  text" in [the algorithm notes](./live-text-algorithm.md) if I ever
  write that doc).

## Known gotchas

- The host's `transkey-web/admin/tsconfig.json` was missing at session
  start (Nuxt 4 scaffold file that had been deleted). Restored with the
  references-style config and committed. If a future session sees it
  missing again, run `node_modules/.bin/nuxi prepare` and re-create
  with the contents at commit `f0ef61f`.
- `tflite_flutter` ships GPU delegate references R8 trips on for
  release builds. The proguard rules in
  `android/app/proguard-rules.pro` `-dontwarn org.tensorflow.lite.gpu.**`
  + `-keep class org.tensorflow.lite.**`. Don't remove these.
- Local migration runner (`npm run migrate`) uses the compiled `dist/`
  output; for new migrations added in the same session, run
  `npm run migrate:dev` (ts-node) OR apply the SQL directly + insert a
  `migrations` row. The `0041-vision-usage.sql` migration was applied
  directly to the local DB this way.
